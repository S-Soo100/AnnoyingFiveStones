# Don'ts — 프로젝트 전역 금지 규칙

> 반복되는 실수를 방지하기 위한 하드 룰. 이 파일은 60줄 이내로 유지하고, 기능별 세부 룰은 `donts/` 하위 파일로 분리한다.

## 🔁 새 룰 추가 기준 (Three-Strike Rule)

| 발생 횟수 | 조치 |
|---------|------|
| **1회** | `~/.claude/projects/.../memory/feedback_*.md` 메모리에만 저장 |
| **2회째** | `.claude/donts-audit.md`에 **"승격 후보"** 플래그 + 맥락 메모 |
| **3회째** | 즉시 이 파일 또는 `donts/{기능}.md`에 정식 룰로 추가 + 메모리에 "donts 승격됨" 표시 |

**재발 판정:** 동일 feedback 파일 재트리거뿐 아니라 **비슷한 맥락**(다른 파일·다른 기능이어도 원인·패턴이 같음)도 포함.
**예외 — 1회에도 즉시 추가:** 사용자가 "다시는 하지 마"라고 명시 / 되돌리기 비용이 매우 큰 사고(DB 삭제, 빌드 파괴 등).
승격 시 등록 위치: 2개 이상 기능에 공통이면 이 파일, 단일 기능 한정이면 `donts/{기능}.md`.

## 🚫 제너럴 (모든 작업)

1. **기억으로 단정 금지** — 에셋·코드·설정을 언급하기 전에 반드시 도구로 확인. 특히 시각 에셋 색상/형태는 `Read`로 열어본 뒤에만 말한다.
2. **최소 변경 원칙** — 요청 범위 밖의 리팩토링/스타일/과잉 수정 금지. "1~3줄로 끝낼 수 있나?"부터 점검.
3. **진단 없는 수정 금지** — 버그 보고 받으면 "콘솔 로그 + 코드 추적 + git diff" 3단 진단 → 그다음에 수정. 추측으로 코드부터 고치지 않는다.
4. **자체 검수 전 "테스트해보세요" 금지** — 수정 후 흐름 추적 + 엣지케이스 점검을 직접 한 뒤에만 사용자에게 넘긴다.
5. **대규모 변경을 통째로 읽지 않기** — 변경 파일 10개+는 `git diff --stat` → 그룹핑 → 순차 처리. 디렉토리 통째 읽기 금지.
6. **같은 모델 자기 리뷰 금지** — 검수는 반드시 다른 모델 교차 리뷰 (Gemini ↔ Claude ↔ Codex). 자기 동의 편향 때문.
7. **기존 구조 보존 기본값** — 맵·동선·UX·연출은 "바꿔달라"고 명시하지 않은 한 유지. 전면 재작성은 합의 후에만.
8. **AI 생성 기획 맹신 금지** — 외부 AI(Gemini/Codex 등)가 만든 기획서는 초안. 반드시 비판적 검수 후 선택 반영.
9. **Claude Code 안에서 `npm run build` 금지** — 리소스 경합으로 세션 불안정. 타입 체크는 `tsc --noEmit`, 실제 빌드는 사용자 터미널에서.
10. **파괴적 git 작업 금지** — `reset --hard`, `push --force`, `branch -D`, `clean -f`는 사용자 명시 승인 없이 금지. 먼저 원인 조사.
11. **병렬 에이전트 경계 준수** — 동시 투입 시 각자 "새 파일만" 쓰게 하고, 같은 파일 수정 금지. 끝나면 자체 검수.
12. **Claude.ai 구독 모드 유지** — API key 모드 전환 금지 (OAuth MCP 연결 끊김).
